关键决策与对应技术

  • 为什么用 Agent 而不是单次 prompt — 需求转用例需要"先检索相似历史 + 分步推理",一次性 prompt 易遗漏和幻觉;ReAct 循环可分步、按需调工具、基于真实检索结果生成,更准更可控。
  • RAG 60%→88% 怎么做到的 — 调 chunk size(400→600 字 + 100 字 overlap)→ 召回率 +12 pt;BM25 + 向量混合检索 → +8 pt;Cross-Encoder Rerank Top-20→Top-3 → +5 pt;Few-shot prompt → +3 pt。控变量在固定 80 条测试集上测的。
  • 为什么用 LoRA 不全量微调 — Qwen2.5-7B 全量需 4×A100、且易过拟合;LoRA r=8 单卡 24GB 跑通,产出 50MB 适配器可热插拔,效果接近全量。
  • 为什么用 vLLM — PagedAttention 显存利用率高、并发吞吐比 transformers 原生快 ~5×,生产服务必选。

核心代码 · ReAct Agent 工具注册

# tools.py · 三类工具按 Tool Calling 协议暴露给 Agent
from langchain.agents import Tool
from langchain_community.vectorstores import Chroma

retriever = Chroma(persist_directory="./vec", embedding_function=emb).as_retriever(
    search_kwargs={"k": 20}        # 先召回 Top-20，下游 rerank 到 Top-3
)

def rag_search(query: str) -> str:
    docs = retriever.get_relevant_documents(query)
    reranked = cross_encoder.rerank(query, docs, top_n=3)
    return "\n---\n".join(d.page_content for d in reranked)

tools = [
    Tool("history_search", rag_search,
         "检索历史相似需求和已有用例。当用户需求涉及已有模块/术语时优先调用。入参=完整需求描述。"),
    Tool("usecase_generate", gen_usecase,
         "基于已检索的上下文生成结构化用例 JSON。仅在已有充分历史参考后调用。"),
    Tool("mcp_internal", call_mcp,
         "调用内部 MCP Server(术语库/合规规则)。当输出需要校验术语合规时调用。"),
]
agent = create_react_agent(llm=qwen_via_vllm, tools=tools, prompt=react_prompt)

常见追问 · 我的回答

为什么用 LoRA 而不是全量微调?

全量微调 Qwen2.5-7B 需要 4×A100,显存和成本极高、还容易过拟合和灾难性遗忘。LoRA 只训练外挂的两个低秩矩阵(我用 r=8、α=16),24GB 单卡就能跑,产出适配器约 50MB、可热插拔。在我们这种领域单一、样本量两千多条的场景下,LoRA 效果接近全量,性价比明显更高。

RAG 准确率 60% → 88% 是怎么测的?

固定一个 80 条的测试集(真实需求-标准用例配对),优化前跑一遍记录"完全匹配 / 部分匹配 / 错误"三档,优化后同一测试集再跑。88% 指"完全匹配 + 可接受的部分匹配"之和。控制变量保证可比——只动检索/Prompt,不动测试集。

Agent 怎么决定调用哪个 Tool?

LangChain 把每个工具的 name + description 拼到 ReAct prompt 里,模型在 Thought 阶段根据当前任务自己选。所以"工具描述写得好不好"直接影响选对率——这也是我后期重点优化的地方:每个工具描述里写明"适用场景"和"不适用场景",并给一两个 few-shot 触发例。

为什么用 LangGraph 不用简单的顺序代码?

项目 2 的流程有分支和异常路径(检索为空走兜底、LLM 超时走降级),用 if-else 会很快变成"if 嵌套地狱"。LangGraph 的状态图把节点和条件边显式声明,可观测性强,加新分支只改图、不动主流程。代码可维护性高一个量级。

怎么防止大模型幻觉?

四层:① RAG 提供真实依据,prompt 里强制"仅基于检索内容回答";② 低 temperature(0.2);③ 结构化输出 + JSON Schema 校验,不合规自动重试;④ 对关键结论挂 citation 原文,人可回溯。

项目 2 是团队项目,你具体负责哪一块?

我负责 RAG 检索链路(Embedding 选型、Chroma 持久化、混合检索调权)和 LangGraph 状态机的节点/兜底分支设计;另外做了 20+ 用户的使用手册和培训,每周收集反馈做 case 复盘。LLM 微调和前端是另一位同学主导,我了解他们的工作但不会硬讲细节。
